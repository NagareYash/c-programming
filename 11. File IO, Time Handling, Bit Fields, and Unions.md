# File I/O, Time Handling, Bit Fields, and Unions

## File I/O Basics

C provides file operations through `<stdio.h>` — the same header we've been using all along for `printf` and `scanf`. In fact, C treats files and the keyboard/screen the same way: both are **streams** of bytes.

### Predefined Streams

You're already familiar with three streams that are always open when your program starts:

|Stream|Purpose|
|---|---|
|`stdin`|Standard input — usually the keyboard|
|`stdout`|Standard output — usually the screen|
|`stderr`|Standard error — also the screen, but kept separate from `stdout` for error messages|

When you call `printf("hello")`, you're really writing to `stdout`. When you call `scanf`, you're reading from `stdin`. File I/O is the same idea — you're just working with **your own** streams instead of the predefined ones.

### The `FILE *` Type

To work with a file, you need a **file pointer** — a variable of type `FILE *` that represents your connection to that file:

```c
FILE *fp;
```

`FILE` is an opaque type — you don't need to know what's inside it. The library handles all the bookkeeping for you. You just pass `fp` to file functions and they know which file you mean.

### `fopen` — Opening a File

To open a file, use `fopen`:

```c
FILE *fopen(const char *filename, const char *mode);
```

`fopen` returns a `FILE *` if it succeeded, or `NULL` if it failed (file doesn't exist, no permission, disk full, etc.).

**Example:**

```c
FILE *fp = fopen("data.txt", "r");
if (fp == NULL) {
    printf("Could not open file!\n");
    return 1;
}
```

**Always check for `NULL`** — if `fopen` failed and you try to use `fp`, your program will crash.

### File Modes

The mode string tells `fopen` what you want to do with the file:

| Mode | Meaning | If file exists | If file doesn't exist |
| ---- | ------- | -------------- | --------------------- |
| `"r"` | Read | Opens it | **Fails** (returns NULL) |
| `"w"` | Write | **Erases it** and starts fresh | Creates a new empty file |
| `"a"` | Append | Opens it, writes go to the end | Creates a new empty file |
| `"r+"` | Read + Write | Opens it (no erase) | **Fails** |
| `"w+"` | Read + Write | **Erases it** | Creates a new empty file |
| `"a+"` | Read + Append | Opens it, writes go to the end | Creates a new empty file |

⚠️ **Be careful with `"w"`** — it **erases the file's contents** the moment you open it. If you opened your only copy of an important file with `"w"` instead of `"r"`, your data is gone.

### Binary Modes

For **binary files** (storing raw bytes — used for structures, images, executables), add a `b` to the mode:

| Text mode | Binary mode        |
| --------- | ------------------ |
| `"r"`     | `"rb"`             |
| `"w"`     | `"wb"`             |
| `"a"`     | `"ab"`             |
| `"r+"`    | `"rb+"` or `"r+b"` |

We'll use binary modes when we get to writing structures to files.

### `fclose` — Closing a File

When you're done with a file, **always close it**:

```c
fclose(fp);
```

This:

1. **Flushes** any pending writes to disk (writes are buffered for speed)
2. Releases the file handle so other programs can use it
3. Frees internal resources

Forgetting to `fclose` can lead to **lost data** (writes never flushed) and **resource leaks**.

## Text File I/O

Text files store data as **human-readable characters**, just like what you see in Notepad. C provides several functions for reading and writing text files — most are direct cousins of functions you already know.

### `fprintf` — Formatted Write to a File

`fprintf` works exactly like `printf`, except its first argument is a `FILE *` instead of going to the screen:

```c
int fprintf(FILE *stream, const char *format, ...);
```

**Example:**

```c
FILE *fp = fopen("output.txt", "w");
if (fp == NULL) return 1;

fprintf(fp, "Name: %s\n", "Alice");
fprintf(fp, "Age: %d\n", 25);
fprintf(fp, "GPA: %.2f\n", 3.85f);

fclose(fp);
```

After this runs, `output.txt` contains:

```text
Name: Alice
Age: 25
GPA: 3.85
```

### `fscanf` — Formatted Read from a File

`fscanf` is `scanf`'s file cousin:

```c
int fscanf(FILE *stream, const char *format, ...);
```

**Example:**

```c
FILE *fp = fopen("data.txt", "r");
if (fp == NULL) return 1;

int age;
float gpa;
char name[50];

fscanf(fp, "%s %d %f", name, &age, &gpa);

printf("Read: %s, %d, %.2f\n", name, age, gpa);
fclose(fp);
```

### `fgets` — Read a Line at a Time

We met `fgets` back in Lab 7 for reading from `stdin`. Its real purpose is reading from any file:

```c
char *fgets(char *str, int n, FILE *stream);
```

`fgets` reads up to `n-1` characters or until a newline (whichever comes first), and stores it in `str`. It's the **safest** way to read text files because it won't overflow your buffer.

**Reading a file line-by-line:**

```c
FILE *fp = fopen("notes.txt", "r");
if (fp == NULL) return 1;

char line[256];
while (fgets(line, sizeof(line), fp) != NULL) {
    printf("%s", line);   // line already includes the newline
}

fclose(fp);
```

`fgets` returns `NULL` when it reaches end-of-file, which is what makes the loop terminate.

### `fputs` — Write a String

`fputs` writes a string to a file **without** adding a newline:

```c
int fputs(const char *str, FILE *stream);
```

```c
fputs("Hello, World!\n", fp);   // newline is part of the string
```

### `fgetc` and `fputc` — One Character at a Time

For character-by-character work:

```c
int c = fgetc(fp);     // reads one character (returns int so it can return EOF)
fputc('A', fp);        // writes one character
```

### `feof` — End-of-File Check

`feof(fp)` returns non-zero when the file's end-of-file indicator has been set. **Important:** the indicator only gets set **after** you try to read past the end — so `feof` is best used **after** a read attempt, not before:

```c
char c;
while ((c = fgetc(fp)) != EOF) {
    putchar(c);
}
```

The pattern `while (fgetc(fp) != EOF)` is more idiomatic than checking `feof` directly.

### Summary — Text File Functions

| Function | What It Does | Like... |
| -------- | ------------ | ------- |
| `fprintf(fp, ...)` | Formatted write | `printf` but to a file |
| `fscanf(fp, ...)` | Formatted read | `scanf` but from a file |
| `fputs(str, fp)` | Write a string | `puts` but to a file (no auto-newline) |
| `fgets(str, n, fp)` | Read a line | safe, won't overflow |
| `fputc(c, fp)` | Write one character | |
| `fgetc(fp)` | Read one character | returns `EOF` at end-of-file |
| `feof(fp)` | Check end-of-file flag | non-zero after a failed read |

## Binary File I/O

Text files are great for human-readable data, but they have two big problems when storing complex data like structures:

1. **Inefficient** — writing an `int` like `12345` as text takes 5 bytes (one per digit), but the actual integer in memory is only 4 bytes
2. **Awkward for structures** — you'd have to format every field with `fprintf`, then carefully `fscanf` it back, hoping the format matches

**Binary files** solve both problems. Instead of converting data to text, binary I/O writes the **raw bytes** of memory directly to disk. When you read it back, you get those same bytes back into memory — no parsing, no formatting.

### `fwrite` — Write Raw Bytes

```c
size_t fwrite(const void *ptr, size_t size, size_t count, FILE *stream);
```

| Parameter | Meaning                        |
| --------- | ------------------------------ |
| `ptr`     | Pointer to the data to write   |
| `size`    | Size of each element in bytes  |
| `count`   | Number of elements to write    |
| `stream`  | The file to write to           |

`fwrite` returns the number of elements actually written (should equal `count` on success).

**Example — writing an integer:**

```c
int x = 42;
FILE *fp = fopen("data.bin", "wb");
fwrite(&x, sizeof(int), 1, fp);    // write 1 element of size sizeof(int)
fclose(fp);
```

**Example — writing an array of integers:**

```c
int arr[5] = {10, 20, 30, 40, 50};
FILE *fp = fopen("data.bin", "wb");
fwrite(arr, sizeof(int), 5, fp);   // write 5 elements of size sizeof(int) each
fclose(fp);
```

The total bytes written is always `size * count`. In the second example, that's `4 * 5 = 20` bytes.

### `fread` — Read Raw Bytes

```c
size_t fread(void *ptr, size_t size, size_t count, FILE *stream);
```

The parameters mirror `fwrite` exactly. `fread` returns the number of elements successfully read — useful for detecting end-of-file or partial reads.

**Example — reading the integer back:**

```c
int x;
FILE *fp = fopen("data.bin", "rb");
fread(&x, sizeof(int), 1, fp);
printf("Read: %d\n", x);   // 42
fclose(fp);
```

The pattern is symmetric: write with `fwrite`, read back the **exact same bytes** with `fread` using identical `size` and `count`.

### Text vs Binary — When to Use Each

| | Text File | Binary File |
| --- | --- | --- |
| Readable in Notepad? | ✓ Yes | ✗ No (looks like garbage) |
| Storage size | Larger (every digit becomes a byte) | Smaller (raw bytes only) |
| Easy for structures? | Awkward — need format strings | **Trivial** — `fwrite` the whole thing |
| Use for | Logs, config files, exported reports | Databases, save files, structures |
| File extension convention | `.txt`, `.csv`, `.log` | `.dat`, `.bin` |

For your project, **binary files are the right choice** for persistent data like books, employees, or parking records.

## Reading and Writing Structures to Files

This is the **single most important pattern** in this entire lab. It's how every project in your final assignment will save its data.

### Writing One Structure to a File

Since `fwrite` works with raw bytes, you can write an entire struct in **one call** by passing its address:

```c
typedef struct {
    char name[50];
    int  roll;
    float gpa;
} Student;

int main() {
    Student s = { .name = "Alice", .roll = 101, .gpa = 3.85f };

    FILE *fp = fopen("student.dat", "wb");
    if (fp == NULL) return 1;

    fwrite(&s, sizeof(Student), 1, fp);   // write 1 Student struct

    fclose(fp);
    return 0;
}
```

That's it. One line. The entire student — name, roll, GPA — is now saved to disk as raw bytes.

### Reading One Structure Back

Reading is just as simple — symmetric to writing:

```c
Student s;

FILE *fp = fopen("student.dat", "rb");
if (fp == NULL) return 1;

fread(&s, sizeof(Student), 1, fp);

printf("Name: %s, Roll: %d, GPA: %.2f\n", s.name, s.roll, s.gpa);

fclose(fp);
```

After `fread`, the `Student` variable `s` is fully populated — the bytes from disk landed exactly where they belong because `Student` has the same memory layout as when we wrote it.

### Writing an Array of Structures

This is where binary file I/O really shines. To save an entire array of structs, just pass the array's base address and the count:

```c
Student roster[3] = {
    { .name = "Alice",   .roll = 101, .gpa = 3.85f },
    { .name = "Bob",     .roll = 102, .gpa = 3.42f },
    { .name = "Charlie", .roll = 103, .gpa = 3.91f }
};

FILE *fp = fopen("roster.dat", "wb");
fwrite(roster, sizeof(Student), 3, fp);   // write all 3 in one call
fclose(fp);
```

One call writes the entire roster. Compare this to the alternative — using `fprintf` to write each field of each student individually. The binary approach is shorter, faster, and harder to get wrong.

### Reading an Array of Structures Back

Same pattern in reverse:

```c
Student roster[3];

FILE *fp = fopen("roster.dat", "rb");
fread(roster, sizeof(Student), 3, fp);
fclose(fp);

for (int i = 0; i < 3; i++) {
    printf("%s - %d - %.2f\n", roster[i].name, roster[i].roll, roster[i].gpa);
}
```

### What If You Don't Know How Many Records to Read?

When you don't know how many structs are in the file ahead of time, use `fread`'s **return value**  it tells you how many elements it successfully read. When it returns 0, you've hit the end:

```c
Student s;
FILE *fp = fopen("roster.dat", "rb");

while (fread(&s, sizeof(Student), 1, fp) == 1) {
    printf("%s - %d - %.2f\n", s.name, s.roll, s.gpa);
}

fclose(fp);
```

This loop reads structs one at a time until the file runs out. It works for any file size — perfect when you're appending records over multiple program runs.

### ⚠️ A Caveat — Pointer Members

The simple `fwrite(&struct, ...)` approach **only works** for structs whose members are all "plain old data" — `int`, `float`, `char[]`, nested structs, etc. If a struct contains a **pointer** member, `fwrite` will save the pointer's address (which will be meaningless next time you run the program). For structs with dynamic members, you'd have to write a custom save function that handles each field separately. For your project, you can almost always avoid this by using **fixed-size arrays** inside your structs (like `char name[50]`) instead of `char *`.

### Summary — The Core Pattern

| Operation | One Struct | Many Structs |
| --------- | ---------- | ------------ |
| Write | `fwrite(&s, sizeof(S), 1, fp)` | `fwrite(arr, sizeof(S), n, fp)` |
| Read | `fread(&s, sizeof(S), 1, fp)` | `fread(arr, sizeof(S), n, fp)` |

Memorize this — it's the foundation of saving data in C, and it's exactly what your project needs.

## File Positioning

Every open file has a **position indicator** — an internal "cursor" that tracks where the next read or write will happen. So far, we've worked sequentially: open a file, read or write from start to end, close. But sometimes you need to **jump around** — to update a specific record, re-read the file from the start, or check the file's size.

### `fseek` — Move to a Specific Position

```c
int fseek(FILE *stream, long offset, int whence);
```

| Parameter | Meaning                          |
| --------- | -------------------------------- |
| `stream`  | The file to reposition           |
| `offset`  | Number of bytes to move          |
| `whence`  | Reference point for the offset   |

`whence` is one of three constants from `<stdio.h>`:

| Constant | Reference Point |
| -------- | --------------- |
| `SEEK_SET` | Beginning of the file |
| `SEEK_CUR` | Current position |
| `SEEK_END` | End of the file |

**Examples:*

```c
fseek(fp, 0, SEEK_SET);     // jump to the beginning
fseek(fp, 100, SEEK_SET);   // jump to byte 100 from the start
fseek(fp, -50, SEEK_END);   // jump to 50 bytes before the end
fseek(fp, 20, SEEK_CUR);    // skip forward 20 bytes from where we are
```

### Use Case — Updating a Specific Record

Say you've saved 10 `Student` structs in a binary file and want to update student #5's GPA. You don't need to rewrite the whole file — you can `fseek` directly to that record:

```c
fseek(fp, 4 * sizeof(Student), SEEK_SET);   // jump to record index 4 (the 5th)
fwrite(&updated_student, sizeof(Student), 1, fp);
```

This works because every `Student` is the same fixed size, so the position of record `i` is always `i * sizeof(Student)` bytes from the start. This is one of the biggest advantages of binary files over text files — fixed-size records make random access trivial.

### `rewind` — Jump Back to the Start

`rewind(fp)` is a shortcut for `fseek(fp, 0, SEEK_SET)`:

```c
rewind(fp);    // back to the beginning
```

Use it when you need to read a file twice in the same program — once to count records, then again to process them.

### `ftell` — Where Am I Right Now?

`ftell(fp)` returns the current position as a `long`:

```c
long pos = ftell(fp);
printf("Current position: %ld bytes from start\n", pos);
```

A common trick uses `ftell` with `fseek(fp, 0, SEEK_END)` to get the **size of a file**:

```c
fseek(fp, 0, SEEK_END);     // jump to the end
long size = ftell(fp);      // current position = file size in bytes
rewind(fp);                  // go back to the start to actually read it
```

For a binary file of `Student` structs, dividing by `sizeof(Student)` gives you the record count without having to track it separately.

### Summary — Positioning Functions

| Function | What It Does |
| -------- | ------------ |
| `fseek(fp, offset, whence)` | Move to a specific position |
| `rewind(fp)` | Jump back to the start |
| `ftell(fp)` | Get the current position |

## Time Handling

Many real-world programs need to work with **time** — recording when something happened, calculating durations, scheduling events, computing fines for overdue books, charging parking fees by hour, etc. C provides time utilities through `<time.h>`.

### The `time_t` Type

`time_t` is a special type that represents a **timestamp** — specifically, the number of seconds elapsed since **January 1, 1970 at 00:00:00 UTC** (this date is called the **Unix epoch**). It's typically a `long` integer.

```c
#include <time.h>

time_t now;
```

### `time()` — Get the Current Time

```c
time_t time(time_t *t);
```

`time()` returns the current timestamp. You can either pass `NULL` and use the return value, or pass a pointer and have it stored there:

```c
time_t now = time(NULL);   // simplest form — what we'll usually use
printf("Seconds since 1970: %ld\n", (long)now);
```

You've actually seen this before — back in Lab 9 we used `srand(time(NULL))` to seed the random number generator. The current time was just being used as a "random enough" starting value.

### `difftime()` — Calculate Duration

```c
double difftime(time_t end, time_t start);
```

`difftime` returns the number of **seconds** between two timestamps as a `double`:

```c
time_t start = time(NULL);

// ... do some work ...

time_t end = time(NULL);
double seconds = difftime(end, start);

printf("That took %.2f seconds\n", seconds);
```

This is the standard pattern for measuring how long something took or how much time passed between two events.

### `ctime()` — Convert to a Readable String

A raw `time_t` (like `1744569600`) isn't very useful to print. `ctime()` converts a timestamp into a **human-readable string**:

```c
char *ctime(const time_t *timer);
```

```c
time_t now = time(NULL);
printf("Right now: %s", ctime(&now));
// Output: Right now: Mon Apr 13 14:23:45 2026
```

`ctime` returns a string that already includes a newline at the end, so you don't need to add one in your `printf`.

### `localtime()` — Break Down Into Components

If you need individual fields (year, month, day, hour, etc.), use `localtime()`:

```c
struct tm *localtime(const time_t *timer);
```

It returns a pointer to a `struct tm` with these members:

| Member | Meaning | Range |
| ------ | ------- | ----- |
| `tm_year` | Years since 1900 | (add 1900 to get the actual year) |
| `tm_mon` | Months since January | 0–11 (add 1 for human-readable) |
| `tm_mday` | Day of the month | 1–31 |
| `tm_hour` | Hours since midnight | 0–23 |
| `tm_min` | Minutes | 0–59 |
| `tm_sec` | Seconds | 0–60 |
| `tm_wday` | Day of week | 0 (Sunday) to 6 |

**Example:**

```c
time_t now = time(NULL);
struct tm *t = localtime(&now);

printf("Date: %d-%02d-%02d\n",
       t->tm_year + 1900,    // years since 1900 → actual year
       t->tm_mon + 1,        // 0-based month → 1-based
       t->tm_mday);

printf("Time: %02d:%02d:%02d\n", t->tm_hour, t->tm_min, t->tm_sec);
```

The `%02d` format specifier zero-pads to 2 digits, so `9:5` becomes `09:05`.

### Summary — Time Functions

| Function | Purpose |
| -------- | ------- |
| `time(NULL)` | Get current timestamp as `time_t` |
| `difftime(end, start)` | Get duration in seconds (as `double`) |
| `ctime(&t)` | Convert timestamp to readable string |
| `localtime(&t)` | Break timestamp into year/month/day/hour fields |

## Bit Fields

A **bit field** is a struct member with a specified width in **bits** rather than bytes. Bit fields let you pack multiple small values into a single integer, saving memory when you have lots of true/false flags or values that only need a few bits.

### The Memory Problem

Suppose you're writing firmware for a microcontroller that has only 256 bytes of RAM. You need to track several true/false states:

```c
typedef struct {
    unsigned char power_on;
    unsigned char wifi_connected;
    unsigned char battery_low;
    unsigned char error_state;
} DeviceStatus;
```

Each `unsigned char` takes **1 full byte** (8 bits), but each variable only stores **1 bit** of actual information. That's 32 bits used to store 4 bits — 87% wasted. On a desktop computer this is fine, but on a microcontroller with 256 bytes total, every byte counts.

### The Bit Field Solution

```c
typedef struct {
    unsigned int power_on        : 1;
    unsigned int wifi_connected  : 1;
    unsigned int battery_low     : 1;
    unsigned int error_state     : 1;
} DeviceStatus;
```

The `: 1` syntax says "this member uses **1 bit**." All four flags now share a **single integer**, dropping memory usage from 4 bytes to (typically) 4 bytes for the whole struct — but with room for 28 more flags inside the same space.

### Syntax

```c
struct {
    type member_name : width;
};
```

- **type** — must be an integer type (`unsigned int`, `int`, `unsigned char`, etc.)
- **width** — number of bits, must be ≤ the bit width of the type
- **member_name** — name of the field (can be omitted to create unnamed padding)

### Bit Fields with More Than One Bit

A bit field can hold any range that fits in its width. To store a value 0–7, you need 3 bits:

```c
typedef struct {
    unsigned int day   : 5;    // 0–31, needs 5 bits
    unsigned int month : 4;    // 0–15, needs 4 bits
    unsigned int year  : 12;   // 0–4095, needs 12 bits
} CompactDate;
```

This packs an entire date into 21 bits — about 3 bytes. A regular `struct` with three `int` members would take 12 bytes.

**Comparison:**

```c
struct date_normal {
    unsigned int d;
    unsigned int m;
    unsigned int y;
};
// sizeof(struct date_normal) = 12 bytes

struct date_compact {
    unsigned int d : 5;
    unsigned int m : 4;
    unsigned int y : 12;
};
// sizeof(struct date_compact) = 4 bytes (typically)
```

### Using Bit Fields

Once defined, bit fields are accessed exactly like normal struct members:

```c
DeviceStatus status;
status.power_on = 1;
status.battery_low = 0;

if (status.power_on) {
    printf("Device is on\n");
}
```

The compiler handles all the bit manipulation behind the scenes. From your code's perspective, they look just like regular variables.

### Rules and Restrictions

| Rule | Why |
| ---- | --- |
| **No pointers to bit fields** (`&field` is illegal) | Bit fields may not start at a byte boundary |
| **No arrays of bit fields** | Same reason — arrays need addressable elements |
| **Bit fields only inside structs/unions** | They're a struct-level feature |
| **Width must fit the type** | Can't have a 33-bit field if the type is `int` |

### Where Bit Fields Are Actually Used

- **Microcontroller registers** — each bit in a hardware register controls a specific pin or feature
- **Network protocols** — packed packet headers (IP, TCP) where every bit has meaning
- **Compressed file formats** — flags packed tightly to save space
- **Embedded systems** — anywhere RAM is precious

You won't use bit fields in your final project (the projects work fine with regular structs), but it's important to know they exist — you'll see them constantly if you ever work with embedded systems or low-level networking code.

## Unions

A **union** is similar to a struct, but with one big difference: in a struct, every member has its **own memory location**. In a union, **all members share the same memory location**. Only one member can hold a meaningful value at a time.

### Syntax

```c
union tag_name {
    type member1;
    type member2;
    // ...
};
```

You can also `typedef` a union just like a struct:

```c
typedef union {
    int i;
    float f;
    char str[20];
} Variant;
```

### How Memory Is Shared

For a union, the compiler reserves enough space for the **largest** member. All members start at the same address:

```text
Memory layout for Variant:
┌────────────────────────────────────┐
│           20 bytes                 │
└────────────────────────────────────┘
  ↑       ↑
  │       │
  │       └── i (4 bytes), f (4 bytes), str (20 bytes) all start here
  │
  └── Same address for every member
```

### Example

```c
#include <stdio.h>

typedef union {
    int i;
    float f;
    char str[20];
} Variant;

int main() {
    Variant v;

    v.i = 42;
    printf("As int: %d\n", v.i);

    v.f = 3.14f;          // overwrites v.i — they share memory!
    printf("As float: %.2f\n", v.f);

    printf("As int now: %d\n", v.i);   // garbage — the bytes were reinterpreted
    return 0;
}
```

**Output:**

```shell
As int: 42
As float: 3.14
As int now: 1078523331
```

After writing to `v.f`, reading `v.i` no longer gives `42` — the same bytes are now being **reinterpreted** as the bit pattern of a float. Whichever member you wrote to last is the one that's "valid."

### When to Use Unions

The most common use of unions is **representing one value that could be one of several types** — without wasting memory on all of them at once. For example, a config setting that might be a number, a string, or a boolean:

```c
typedef enum { TYPE_INT, TYPE_FLOAT, TYPE_STRING } ValueType;

typedef struct {
    ValueType type;
    union {
        int i;
        float f;
        char str[50];
    } value;
} Setting;
```

This is called a **tagged union** — the `type` field tells you which member of the union is currently valid. You'd never write to `value.i` and then read `value.f`, because that's nonsense — but if `type == TYPE_INT`, you know `value.i` is the one to use.

### Bit Fields + Unions in Microcontrollers

Where unions really earn their keep is in microcontroller programming — combined with bit fields. You can define the same hardware register two ways: as a single byte (for fast operations) and as individual control bits (for clarity):

```c
typedef union {
    unsigned char byte;       // access all 8 bits at once
    struct {
        unsigned char led    : 1;
        unsigned char motor  : 1;
        unsigned char buzzer : 1;
        unsigned char unused : 5;
    } bits;                   // access individual bits by name
} ControlRegister;

ControlRegister reg;
reg.byte = 0;          // clear everything
reg.bits.led = 1;      // turn on the LED
reg.bits.motor = 1;    // turn on the motor
```

Both `reg.byte` and `reg.bits` refer to **the exact same memory** — you just choose which view is more convenient at the moment.

### Struct vs Union — Side by Side

| | Struct | Union |
| --- | --- | --- |
| Members stored | **Each in its own location** | **All share the same location** |
| Total size | Sum of all members (+ padding) | Size of the **largest** member |
| Members usable at once | All | Only one |
| Use for | Bundling related but distinct data | One value that could be one of several types |
