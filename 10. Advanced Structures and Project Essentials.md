# Advanced Structures and Project Essentials

## Enums

An **enumeration** (or `enum`) is a user-defined type for a **set of named integer constants**. Instead of remembering that `0` means "car" and `1` means "bike," you can give those numbers meaningful names.

### Why Use Enums?

Imagine writing a parking lot program without enums:

```c
int vehicle_type = 0;        // 0 = car, 1 = bike, 2 = truck ???
int slot_status  = 1;        // 0 = free, 1 = occupied ???
```

This code is **fragile**: you have to remember what each number means, and if you accidentally use `vehicle_type = 5` nothing will stop you. Enums solve this by giving each value a name:

```c
enum VehicleType { CAR, BIKE, TRUCK };
enum SlotStatus  { FREE, OCCUPIED };

enum VehicleType vehicle = CAR;
enum SlotStatus  status  = OCCUPIED;
```

Now the intent is crystal clear.

### Declaration

**Syntax:**

```c
enum tag_name { VALUE1, VALUE2, VALUE3, ... };
```

By default, the first value is `0`, the next is `1`, and so on:

```c
enum Day { MONDAY, TUESDAY, WEDNESDAY, THURSDAY, FRIDAY, SATURDAY, SUNDAY };
//          0        1         2          3         4       5         6
```

You can also assign explicit values:

```c
enum Status { IDLE = 1, RUNNING = 2, STOPPED = 5 };
```

Unspecified values continue from the previous one:

```c
enum Level { LOW = 1, MEDIUM, HIGH, CRITICAL };
//            1        2       3       4
```

### Using Enums

```c
#include <stdio.h>

enum VehicleType { CAR, BIKE, TRUCK };

int main() {
    enum VehicleType v = BIKE;

    if (v == BIKE) {
        printf("It's a bike!\n");
    }

    printf("BIKE = %d\n", BIKE);   // prints: BIKE = 1
    return 0;
}
```

Under the hood, enum values are just **integers** — you can compare them, print them, and use them in arithmetic. But the **names** make your code far more readable.

### `typedef enum` — The Cleaner Way

Just like with structures, you can combine `typedef` with `enum` to avoid writing `enum` everywhere:

```c
typedef enum {
    CAR,
    BIKE,
    TRUCK
} VehicleType;

VehicleType v = CAR;    // no need for 'enum VehicleType'
```

This is the **preferred** style and what we'll use going forward.

### Enums with `switch` Statements

Enums pair beautifully with `switch` — the compiler can even warn you if you forget to handle a case:

```c
typedef enum { CAR, BIKE, TRUCK } VehicleType;

void printVehicle(VehicleType v) {
    switch (v) {
        case CAR:
            printf("Four wheels\n");
            break;
        case BIKE:
            printf("Two wheels\n");
            break;
        case TRUCK:
            printf("Heavy vehicle\n");
            break;
    }
}
```

### Enums Inside Structures

This is where enums really shine. Most real-world structures have a "type" or "status" field, and that's exactly what enums are for:

```c
typedef enum { AVAILABLE, OCCUPIED, RESERVED } SlotStatus;

typedef struct {
    int slot_number;
    SlotStatus status;
    char license_plate[10];
} ParkingSlot;

ParkingSlot s1 = { .slot_number = 42, .status = OCCUPIED, .license_plate = "ABC123" };

if (s1.status == OCCUPIED) {
    printf("Slot %d is taken\n", s1.slot_number);
}
```

Notice how `s1.status == OCCUPIED` reads like plain English. Compare that to `s1.status == 1` — which one would you rather debug?

### Summary

| Feature | Benefit |
| ------- | ------- |
| Named constants | Readable code — `CAR` instead of `0` |
| Type safety (mostly) | Compiler treats enums as their own type |
| Pairs with `switch` | Clean state-based logic |
| Pairs with structures | Natural way to represent status fields |

## Structures Containing Arrays

So far our struct examples have used simple types like `int` and `float`. But struct members can be **anything** — including **arrays**. This is how we model entities with text fields (names, addresses, license plates, titles).

### Example

```c
typedef struct {
    char name[50];
    int  roll;
    float gpa;
} Student;
```

The `name` field is a fixed-size character array of 50 bytes. The struct's total size is the sum of all its members (plus possible padding).

### A Critical Caveat — Arrays Are Constant Pointers

Recall from Lab 5 that an array name is a **constant pointer** to its first element. This means you **cannot assign** to an array using `=`:

```c
Student s;
s.name = "Alice";    // ⚠️ COMPILE ERROR — can't assign to an array
```

To put a string into an array member, you must use `strcpy` from `<string.h>`:

```c
#include <string.h>

Student s;
strcpy(s.name, "Alice");    // ✓ this works
s.roll = 101;
s.gpa  = 3.85f;
```

**The exception:** designated initializers at declaration time work, because the compiler treats them as initialization (not assignment):

```c
Student s = { .name = "Alice", .roll = 101, .gpa = 3.85f };   // ✓ works
```

### Struct Assignment Still Works — Even With Arrays Inside

Here's something interesting: even though you can't assign to a bare array, you **can** copy an entire struct (arrays included) with `=`:

```c
Student s1 = { .name = "Alice", .roll = 101, .gpa = 3.85f };
Student s2;

s2 = s1;     // ✓ copies EVERYTHING — including the entire name array

printf("%s\n", s2.name);    // Alice
```

The compiler treats the whole struct as one block of memory and copies it byte-for-byte. The `name` array gets copied along with everything else. This is a major reason why structs are so convenient compared to passing around several separate variables.

## Structures and Functions — Pass by Value

Since a struct is "just a variable," you can pass it to a function the same way you pass an `int` — by value. The function receives a **complete copy** of the struct.

```c
#include <stdio.h>
#include <string.h>

typedef struct {
    char name[50];
    int  roll;
} Student;

void display(Student s) {
    printf("Name: %s\n", s.name);
    printf("Roll: %d\n", s.roll);
}

int main() {
    Student s1;
    strcpy(s1.name, "Alice");
    s1.roll = 101;

    display(s1);
    return 0;
}
```

This works fine for **reading** struct data, but it has the same limitation as passing any variable by value: **changes inside the function don't affect the original**. If `display` tried to modify `s.roll`, it would only change the local copy.

There's also a **performance cost** — copying a large struct (especially one with big arrays inside) can be slow. For these reasons, passing by value is rarely the right choice for structs.

## Pointers to Structures

To modify a struct from inside a function — or to avoid the cost of copying — we use **pointers to structures**, just like we used pointers to integers in Lab 4.

### Declaration

```c
typedef struct {
    char name[50];
    int  roll;
    float gpa;
} Student;

Student s1;
Student *ptr;     // pointer to a Student

ptr = &s1;        // ptr now points to s1
```

### Accessing Members Through a Pointer

If you have a regular struct variable, you use `.`:

```c
s1.roll = 101;
```

If you have a **pointer** to a struct, you have two options:

**Option 1 — Dereference, then dot:**

```c
(*ptr).roll = 101;    // works, but ugly
```

The parentheses are required because `.` has higher precedence than `*` — without them, `*ptr.roll` would be interpreted as `*(ptr.roll)`, which is wrong.

**Option 2 — The arrow operator `->`:**

C provides a special shortcut, the **arrow operator** (`->`), that does both steps in one:

```c
ptr->roll = 101;      // exactly the same as (*ptr).roll = 101
```

The arrow operator is **always preferred** when working with struct pointers — it's cleaner, easier to read, and the standard idiom in C. Read it as "the `roll` of what `ptr` points to."

### `(*ptr).member` vs `ptr->member`

| Expression | Same Thing? |
| ---------- | ----------- |
| `(*ptr).name` | `ptr->name` |
| `(*ptr).roll` | `ptr->roll` |
| `(*ptr).gpa` | `ptr->gpa` |

Always use `->` for struct pointers. Reserve `(*ptr).member` for the rare case when you want to be explicit about the dereferencing step.

## Structures and Functions — Pass by Reference

Now we can pass a struct **by reference** — which means passing its **address** (`&struct_var`) and receiving it as a pointer (`Struct *`). This lets the function:

1. **Modify** the original struct directly
2. **Avoid the copying overhead** of pass by value

### Example

```c
#include <stdio.h>
#include <string.h>

typedef struct {
    char name[50];
    int  roll;
    float gpa;
} Student;

void updateGPA(Student *s, float new_gpa) {
    s->gpa = new_gpa;       // modifies the original through the pointer
}

void display(Student *s) {
    printf("Name: %s, Roll: %d, GPA: %.2f\n", s->name, s->roll, s->gpa);
}

int main() {
    Student s1 = { .name = "Alice", .roll = 101, .gpa = 3.50f };

    display(&s1);             // pass address of s1
    updateGPA(&s1, 3.85f);    // updates the actual s1
    display(&s1);

    return 0;
}
```

**Output:**

```shell
Name: Alice, Roll: 101, GPA: 3.50
Name: Alice, Roll: 101, GPA: 3.85
```

Notice how the second call to `display` shows the updated GPA — `updateGPA` modified the original through the pointer.

### When to Use Each Approach

| Approach | Syntax | Use When |
| -------- | ------ | -------- |
| Pass by value | `func(s)` / parameter `Student s` | You only need to **read** the struct and it's small |
| Pass by reference | `func(&s)` / parameter `Student *s` | You need to **modify** the struct, OR the struct is **large** (efficiency) |

**Rule of thumb:** Almost always prefer pass by reference for structs. It's faster, more flexible, and matches the conventions used throughout C standard library and real-world code.

## Arrays of Structures

We touched on arrays of structures briefly in Lab 9 (Problem 3 with the `Student` array). Now let's look at them more carefully — they're the **single most common pattern** in real C programs and the foundation of every project you'll build.

### Why Arrays of Structures?

A single struct represents **one** entity — one student, one book, one parking slot. To manage a **collection** of entities, you need an array of those structs. Almost every real-world system is structured this way:

- A library has an **array of `Book` structs**
- A restaurant has an **array of `MenuItem` structs**
- A parking lot has an **array of `ParkingSlot` structs**
- A payroll system has an **array of `Employee` structs**

### Declaration and Access

```c
typedef struct {
    char name[50];
    float price;
    int   stock;
} Product;

Product inventory[5];   // array of 5 Product structs

inventory[0].price = 9.99f;
inventory[1].stock = 42;
strcpy(inventory[2].name, "Coffee Beans");
```

Each element of the array is a complete struct. You access its members using `[index].member`.

### Iterating Through an Array of Structures

A simple `for` loop walks through every struct in the array:

```c
for (int i = 0; i < 5; i++) {
    printf("%s - $%.2f (%d in stock)\n",
           inventory[i].name,
           inventory[i].price,
           inventory[i].stock);
}
```

### Passing an Array of Structures to a Function

Just like with regular arrays (Lab 5), passing an array of structs to a function actually passes a **pointer to the first element**. You should also pass the size separately:

```c
void displayInventory(Product products[], int size) {
    for (int i = 0; i < size; i++) {
        printf("%s - $%.2f\n", products[i].name, products[i].price);
    }
}

int main() {
    Product inventory[5] = { /* ... */ };
    displayInventory(inventory, 5);
}
```

Inside the function, `products[i].name` works exactly like in `main`. Any modification made to `products[i]` inside the function affects the original array — because we're working with a pointer.

### Equivalent Function Signatures

These three are all the same:

```c
void func(Product products[], int size);     // most readable
void func(Product *products, int size);      // pointer notation
void func(Product products[5], int size);    // size in brackets is ignored
```

## Dynamic Arrays of Structures

Static arrays (like `Product inventory[5]`) work fine when you know the size in advance. But what if the user decides at runtime how many items there will be? That's where dynamic memory comes in — combining what we learned in Lab 8 with structures.

### Allocating a Single Struct

You can `malloc` a single structure just like any other type:

```c
Product *p = (Product *)malloc(sizeof(Product));

if (p == NULL) {
    printf("Allocation failed!\n");
    return 1;
}

strcpy(p->name, "Apple");
p->price = 0.99f;
p->stock = 100;

free(p);
```

Notice we use the **arrow operator** (`->`) because `p` is a pointer to a struct. This is the same pattern as Section 2's pass-by-reference.

### Allocating an Array of Structs

To allocate space for `n` structures, multiply `sizeof(struct)` by `n`:

```c
int n;
printf("How many products? ");
scanf("%d", &n);

Product *inventory = (Product *)malloc(n * sizeof(Product));

if (inventory == NULL) {
    printf("Allocation failed!\n");
    return 1;
}
```

Now `inventory` points to a block of memory big enough to hold `n` Product structs. You access them with **either** subscript notation **or** pointer arithmetic:

```c
inventory[0].price = 1.99f;     // subscript notation — clean and clear
(inventory + 1)->price = 2.99f;  // pointer notation — same thing
```

Subscript notation is preferred — it reads better.

### Cleanup

When you're done, `free` the entire block in one call — you don't free each struct individually:

```c
free(inventory);
```

This works because `malloc` allocated **one contiguous block** for all `n` structs.

### Complete Example

```c
#include <stdio.h>
#include <stdlib.h>
#include <string.h>

typedef struct {
    char name[50];
    float price;
} Product;

int main() {
    int n;
    printf("How many products? ");
    scanf("%d", &n);

    Product *inventory = (Product *)malloc(n * sizeof(Product));
    if (inventory == NULL) return 1;

    for (int i = 0; i < n; i++) {
        printf("Enter name and price for product %d: ", i + 1);
        scanf("%s %f", inventory[i].name, &inventory[i].price);
    }

    printf("\nInventory:\n");
    for (int i = 0; i < n; i++) {
        printf("%s - $%.2f\n", inventory[i].name, inventory[i].price);
    }

    free(inventory);
    return 0;
}
```

### Static vs Dynamic Arrays of Structures

| | Static | Dynamic |
| --- | --- | --- |
| Declaration | `Product arr[5];` | `Product *arr = malloc(n * sizeof(Product));` |
| Size known | At compile time | At runtime |
| Access | `arr[i].field` | `arr[i].field` (same!) |
| Cleanup | Automatic | `free(arr);` |
| Use when | Fixed size | User decides size |

The beauty here is that the **access syntax is identical** for both. Once you've allocated the memory, working with a dynamic array of structs feels exactly the same as working with a static one.

## Structures with Dynamically-Sized Internal Arrays

So far, when our structs contained arrays, those arrays had **fixed sizes** (`char name[50]`). But what if the size depends on user input? For example, what if you want a `Team` struct that can hold any number of players?

You can't write:

```c
typedef struct {
    char team_name[50];
    int  num_players;
    char players[num_players][50];   // ⚠️ ILLEGAL — size must be a constant
} Team;
```

The size of an array inside a struct must be **known at compile time**. So instead of putting an array directly in the struct, we put a **pointer** that we'll allocate dynamically at runtime.

### The Pattern

```c
typedef struct {
    char  team_name[50];
    int   num_players;
    char *players;     // pointer to dynamically allocated memory
} Team;
```

Now `players` is a pointer that we can `malloc` to whatever size we need.

### Two-Step Allocation

This is critical and trips up a lot of students: when you allocate the struct, you only get memory for **the struct itself**. The pointer member is just 8 bytes (or 4 on 32-bit systems) — it doesn't yet point to anything valid. You have to **separately** allocate memory for what the pointer should point to.

```c
// Step 1 — allocate the struct itself
Team *t = (Team *)malloc(sizeof(Team));

// Step 2 — allocate memory for the internal pointer member
strcpy(t->team_name, "Engineering");
t->num_players = 5;
t->players = (char *)malloc(t->num_players * 50 * sizeof(char));   // 5 names, 50 chars each
```

If you skip step 2, `t->players` is an **uninitialized pointer** — using it will crash your program.

### Cleanup Order — Inside Out

Just as importantly, when you `free` the struct, you have to free the **internal pointer first**. If you free the struct first, you lose the address of the internal pointer and can never free it — that's a memory leak.

```c
// Cleanup — inside out
free(t->players);    // first: free the internal allocation
free(t);             // second: free the struct itself
```

Think of it like packing up nested boxes: empty the inner ones before throwing away the outer one.

### Visualizing the Two Allocations

```text
Heap memory:

  Team struct (allocated by malloc(sizeof(Team)))
  ┌──────────────────────────────┐
  │ team_name: "Engineering"     │
  │ num_players: 5               │
  │ players: ──────────────────┐ │
  └──────────────────────────────┘
                                │
                                ▼
  Player names (allocated by separate malloc)
  ┌──────────────────────────────┐
  │ Alice...Bob...Charlie...     │
  │ ...Diana...Eve               │
  └──────────────────────────────┘
```

The struct is one block; its `players` member points to a **completely separate block** of memory.

### Complete Example

```c
#include <stdio.h>
#include <stdlib.h>
#include <string.h>

typedef struct {
    char  team_name[50];
    int   num_players;
    char *players;
} Team;

int main() {
    int n;
    printf("How many players? ");
    scanf("%d", &n);

    // Step 1: allocate the struct
    Team *t = (Team *)malloc(sizeof(Team));
    if (t == NULL) return 1;

    strcpy(t->team_name, "Engineering");
    t->num_players = n;

    // Step 2: allocate the internal array
    t->players = (char *)malloc(n * 50 * sizeof(char));
    if (t->players == NULL) {
        free(t);
        return 1;
    }

    // Use it
    for (int i = 0; i < n; i++) {
        printf("Player %d name: ", i + 1);
        scanf("%s", &t->players[i * 50]);   // pointer arithmetic to get to slot i
    }

    // Cleanup — inside out
    free(t->players);
    free(t);

    return 0;
}
```

This pattern is exactly what you'll need in your project for things like a library member's borrowed-books list, a restaurant order's items, or a parking record's logged events.

## Nested Structures

A struct member can be **another struct**. This is called **nesting**, and it's how you model entities that contain other entities — like a restaurant order containing menu items, an employee containing a hire date, or a book containing a publication date.

### Example — Restaurant Order

A restaurant `Order` contains an array of `MenuItem`s. Each `MenuItem` is itself a struct with a name and price:

```c
typedef struct {
    char  name[50];
    float price;
} MenuItem;

typedef struct {
    int       table_number;
    int       item_count;
    MenuItem  items[5];     // array of 5 MenuItem structs INSIDE Order
} Order;
```

Now an `Order` is a complete entity that holds everything about a table's order — the table number, how many items have been ordered, and the actual list of items.

### Accessing Nested Members

To reach a member of a nested struct, just **chain the dot operators**:

```c
Order o;
o.table_number = 5;
o.item_count   = 2;

// Set the first item
strcpy(o.items[0].name, "Burger");
o.items[0].price = 12.99f;

// Set the second item
strcpy(o.items[1].name, "Fries");
o.items[1].price = 4.99f;
```

Read `o.items[0].price` as: "the order's items array, element 0's price."

### Designated Initializers Work for Nested Structs Too

You can initialize nested structs at declaration time using the same designated initializer syntax:

```c
Order o = {
    .table_number = 5,
    .item_count   = 2,
    .items = {
        { .name = "Burger", .price = 12.99f },
        { .name = "Fries",  .price = 4.99f  }
    }
};
```

This is clean, explicit, and self-documenting.

### With Pointers — `->` and `.` Combined

When you have a pointer to an outer struct, use `->` to reach the outer members and `.` to keep going into nested members:

```c
Order *p = &o;

p->table_number = 5;       // -> for the first hop (because p is a pointer)
p->items[0].price = 12.99f; // -> to get into items, then . because items[0] is not a pointer
```

The rule: **use `->` when you're going through a pointer; use `.` when you're working with a struct directly.**

### Why Nested Structs Matter

Nesting lets you model **hierarchies and compositions** naturally. Without it, you'd have to create one giant flat struct with dozens of unrelated fields, or maintain parallel arrays in awkward ways. With nested structs, each layer of the hierarchy is its own clean type that can be reused independently.

For example, the same `MenuItem` struct can appear:

- Inside an `Order` (as part of a customer's bill)
- Inside an array of all menu items the restaurant offers
- Inside a `DailySpecial` struct alongside other metadata

That reusability is one of the reasons real C programs lean so heavily on structures.
