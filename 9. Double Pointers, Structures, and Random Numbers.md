# Double Pointers, Structures, and Random Numbers

## Random Numbers

Many programs need to generate random values — for simulations, games, testing with sample data, or cryptographic applications. C provides random number functionality through `<stdlib.h>`.

### The `rand()` Function

```c
int rand(void);
```

`rand()` returns a **pseudo-random integer** between `0` and `RAND_MAX`. `RAND_MAX` is a constant defined in `<stdlib.h>` — its value is at least `32767` but is often much larger.

**Basic example:**

```c
#include <stdio.h>
#include <stdlib.h>

int main() {
    printf("Random number: %d\n", rand());
    printf("RAND_MAX: %d\n", RAND_MAX);
    return 0;
}
```

### The Problem — Same Numbers Every Run

Try running the program above multiple times. You'll notice something strange: **you get the same number every time!** This is because `rand()` is **pseudo-random** — it uses a mathematical formula that produces a predictable sequence starting from a fixed initial value called the **seed**.

### Seeding with `srand()`

To get different numbers on each run, we seed the random generator with a **different starting value** each time using `srand()`:

```c
void srand(unsigned int seed);
```

But what should we use as a seed? The best choice is the **current time** — which changes every second — available through `time(NULL)` from `<time.h>`:

```c
#include <stdio.h>
#include <stdlib.h>
#include <time.h>

int main() {
    srand(time(NULL));     // seed once, at the start of main

    printf("Random: %d\n", rand());
    printf("Random: %d\n", rand());
    return 0;
}
```

**Important:** Call `srand()` **only once** at the start of your program. If you reseed repeatedly in a tight loop, you might reseed multiple times in the same second and get the same number each time.

### Generating Numbers in a Range

`rand()` returns a number between `0` and `RAND_MAX`. To get a number in a specific range, use the **modulo operator**:

```c
// Random number from 0 to 9
int x = rand() % 10;

// Random number from 1 to 100
int y = (rand() % 100) + 1;

// Random number from min to max (inclusive)
int z = (rand() % (max - min + 1)) + min;
```

**Example — dice roll:**

```c
#include <stdio.h>
#include <stdlib.h>
#include <time.h>

int main() {
    srand(time(NULL));

    printf("Rolling dice 5 times:\n");
    for (int i = 0; i < 5; i++) {
        int roll = (rand() % 6) + 1;     // 1 to 6
        printf("Roll %d: %d\n", i + 1, roll);
    }

    return 0;
}
```

**Output (will differ each run):**

```shell
Rolling dice 5 times:
Roll 1: 4
Roll 2: 2
Roll 3: 6
Roll 4: 1
Roll 5: 5
```

### Summary

| Step | Function                | Purpose                                            |
| ---- | ----------------------- | -------------------------------------------------- |
| 1    | `srand(time(NULL))`     | Seed the generator, call **once** at program start |
| 2    | `rand()`                | Generate the next number in the sequence           |
| 3    | `rand() % n`            | Restrict the number to a range `[0, n-1]`          |
| 4    | `(rand() % n) + offset` | Shift the range to start from `offset`             |

## Double Pointers

In Lab 5 we learned that a pointer stores the **memory address** of a variable. A **double pointer** takes this one step further — it's a pointer that stores the **address of another pointer**.

### Why Would We Need This?

Think back to Lab 4, when we learned pass by reference. If you want a function to modify an `int`, you pass `&x` and the function receives `int *`. But what if you want a function to modify a **pointer itself** — not the value it points to, but *where it points*? A single pointer won't cut it. You need a pointer **to the pointer** — a double pointer.

### Declaration

**Syntax:**

```c
type **pointer_name;
```

The `**` tells the compiler "this is a pointer to a pointer."

**Example:**

```c
int w = 77;          // a regular integer
int *ptr2;           // a regular pointer to int
int **ptr1;          // a double pointer — pointer to a pointer to int

ptr2 = &w;           // ptr2 holds the address of w
ptr1 = &ptr2;        // ptr1 holds the address of ptr2
```

### Visualizing in Memory

```text
Variable    Address      Value
--------    -------      -----
   w        0x1000        77
   ptr2     0x2000      0x1000   ← address of w
   ptr1     0x3000      0x2000   ← address of ptr2
```

- `w` holds the value `77`
- `ptr2` holds the **address of w** (`0x1000`)
- `ptr1` holds the **address of ptr2** (`0x2000`)

### Accessing the Value — Step by Step

To reach `w` through `ptr1`, you need to dereference **twice**:

```c
printf("w      = %d\n", w);           // 77
printf("*ptr2  = %d\n", *ptr2);       // 77 — one dereference: address → value
printf("**ptr1 = %d\n", **ptr1);      // 77 — two dereferences: address → address → value
```

**Breaking down `**ptr1`:**

```text
ptr1     → 0x2000   (address of ptr2)
*ptr1    → 0x1000   (value at 0x2000, which is the address of w)
**ptr1   → 77       (value at 0x1000, which is w itself)
```

Each `*` "steps inward" one level:

| Expression | Result | Meaning |
| ---------- | ------ | ------- |
| `ptr1` | `0x2000` | Address of `ptr2` |
| `*ptr1` | `0x1000` | Same as `ptr2` — the address of `w` |
| `**ptr1` | `77` | Same as `w` — the actual value |

### Why Not Just Use a Single Pointer?

The key insight is **what you want to modify**. Consider the problem of swapping two pointers — not the values they point to, but where they point. With a single pointer, you can't do this inside a function:

**The problem — single pointer doesn't work:**

```c
void bad_swap(int *a, int *b) {
    int *tmp = a;
    a = b;         // only swaps local copies!
    b = tmp;
}

int main() {
    int x = 10, y = 20;
    int *p = &x;   // p points to x
    int *q = &y;   // q points to y

    bad_swap(p, q);

    printf("p points to %d\n", *p);   // still 10!
    printf("q points to %d\n", *q);   // still 20!
}
```

`bad_swap` only received **copies** of `p` and `q`. The original pointers in `main` are untouched. This should look familiar — it's exactly the same issue we saw with pass-by-value in Lab 4.

**The fix — pass pointers to the pointers:**

```c
void swap_pointers(int **a, int **b) {
    int *tmp = *a;
    *a = *b;       // modifies the actual pointer in main!
    *b = tmp;
}

int main() {
    int x = 10, y = 20;
    int *p = &x;
    int *q = &y;

    printf("Before: p → %d, q → %d\n", *p, *q);
    swap_pointers(&p, &q);           // pass addresses of the pointers
    printf("After:  p → %d, q → %d\n", *p, *q);
}
```

**Output:**

```shell
Before: p → 10, q → 20
After:  p → 20, q → 10
```

Now `p` points to `y` and `q` points to `x` — the pointers themselves have been swapped without moving any actual data. This is the same idea we used in Lab 7 when we swapped pointers in an array of strings — we never moved the strings, only the addresses.

### Golden Rule
>
> If a function needs to **modify a pointer variable** in the caller, the function must receive a **pointer to that pointer** (`type **`).

### The Pattern So Far

| What the function modifies | Pass this | Function parameter type |
| -------------------------- | --------- | ----------------------- |
| An `int` | `&x` | `int *` |
| A pointer (where it points) | `&ptr` | `int **` |

This scales indefinitely: to modify a double pointer, you'd pass a triple pointer — though you rarely need to go beyond double pointers in practice.

## Structures

So far, the custom "containers" we've used have been **arrays** — collections of values of the **same** type. But what if you need to group together values of **different** types? For example, a student record might have a name (string), a roll number (int), and a GPA (float). You could use three separate variables, but they'd be loose — nothing ties them together.

A **structure** lets you bundle different types of data into a single custom variable. Structures are the foundation of almost every real-world C program — they're how we model things like students, books, orders, bank accounts, or any "entity" with multiple properties.

### Comparison

| | Regular Variable | Array | Structure |
| --- | --- | --- | --- |
| Holds | One value | Many values | Many values |
| Data types allowed | One | One (all same) | **Any mix** |
| Example | `int age` | `int grades[5]` | A student with name, age, and GPA |

### Defining a Structure

**Syntax:**

```c
struct tag_name {
    type member1;
    type member2;
    // ...
};
```

- `struct` is the keyword
- `tag_name` is the name of this structure type (sometimes called the "tag")
- The members can be **any valid variable type** — `int`, `float`, `char`, arrays, pointers, even other structures

**Example — a date structure:**

```c
struct date {
    int month;
    int day;
    int year;
};
```

**Important:** This definition **does not allocate any memory**. It's just a blueprint telling the compiler "here's what a `date` looks like." Memory is allocated only when you **declare a variable** of this type.

### Declaring Variables of a Structure Type

Once you've defined a structure, you can declare variables of that type:

```c
struct date today;              // one variable
struct date yesterday, tomorrow;  // multiple variables
```

You can also **combine** the definition and declaration in one step:

```c
struct date {
    int month;
    int day;
    int year;
} today;           // today is declared immediately
```

Or declare multiple variables alongside the definition:

```c
struct date {
    int month;
    int day;
    int year;
} today, yesterday, tomorrow;
```

### The `struct` Keyword Gets Annoying

Notice that every time we declare a variable, we have to write `struct date` — not just `date`. In a large program, this gets repetitive fast:

```c
struct date birthday;
struct date anniversary;
struct date graduation;
// ...
```

Wouldn't it be nice if we could just write `date birthday;`? That's what `typedef` does.

### `typedef struct` — The Clean Way

We saw `typedef` back in Lab 4. It lets us create an **alias** for a type. Combining `typedef` with `struct` lets us drop the `struct` keyword entirely when declaring variables.

**Syntax:**

```c
typedef struct {
    type member1;
    type member2;
    // ...
} new_type_name;
```

**Example:**

```c
typedef struct {
    int month;
    int day;
    int year;
} date;

date today, yesterday, tomorrow;   // no need for 'struct' anymore!
```

This is the **cleanest, most common** way to define structures in modern C, and it's what we'll use going forward.

### Accessing Members — The `.` Operator

Once you have a structure variable, you access its members using the **dot operator** (`.`):

```c
date today;

today.month = 4;
today.day   = 13;
today.year  = 2026;

printf("Today is %d/%d/%d\n", today.month, today.day, today.year);
```

**Output:**

```shell
Today is 4/13/2026
```

Each member behaves like a normal variable — you can read it, write to it, pass it to functions, and perform any operation you'd normally do:

```c
typedef struct {
    int j;
    int a;
    int b;
} stuff;

stuff test;
test.a = 10;
test.b = 20;
test.j = test.a + test.b;    // 30 — works just like regular variables
```

### Initialization

Structures can be initialized in several ways.

**Method 1 — Set members one by one:**

```c
date yesterday;
yesterday.month = 4;
yesterday.day   = 12;
yesterday.year  = 2026;
```

**Method 2 — Positional initialization (like arrays):**

```c
date yesterday = {4, 12, 2026};   // month, day, year — in order
```

This works, but has a problem: if you ever **reorder** the fields in your struct definition, your initializers silently become wrong. If you change `date` to put `year` first, `{4, 12, 2026}` now means year 4, month 12, day 2026 — and the compiler won't warn you.

**Method 3 — Designated initializers (the best way):**

C99 introduced **designated initializers**, which let you name each field explicitly:

```c
date yesterday = { .month = 4, .day = 12, .year = 2026 };
```

Now it doesn't matter what order the fields are in the struct definition — each value is tied to its field name by the dot syntax. This is more readable and far safer.

You can even **skip fields** — unspecified fields are set to `0`:

```c
date partial = { .year = 2026 };   // month = 0, day = 0, year = 2026
```

### Why Designated Initializers Are Preferred'

| | Positional | Designated |
| --- | --- | --- |
| Readability | Poor — have to count | Excellent — each value is labeled |
| Reorder-safe? | **No** — breaks silently | **Yes** — order doesn't matter |
| Skip fields? | No (must use `0` placeholder) | Yes |
| Self-documenting? | No | Yes |

Going forward in this lab, we'll use designated initializers whenever we initialize a struct at declaration time.

### A Complete Example

```c
#include <stdio.h>

typedef struct {
    char name[50];
    int  roll;
    float gpa;
} student;

int main() {
    // Method 1: declare then assign
    student s1;
    s1.roll = 101;
    s1.gpa  = 3.85f;
    // (name assignment needs strcpy — we'll see this in a moment)

    // Method 3: designated initializer
    student s2 = { .name = "Jane Doe", .roll = 102, .gpa = 3.92f };

    printf("Student 2: %s, Roll: %d, GPA: %.2f\n", s2.name, s2.roll, s2.gpa);
    return 0;
}
```

**Output:**

```shell
Student 2: Jane Doe, Roll: 102, GPA: 3.92
```

Notice that we initialized `s2.name` with a string literal using designated initializers — this works because we're setting the member **at declaration time**. Assigning a string to a char-array field *after* declaration requires `strcpy` (we'll cover this in Lab 10).

## Copying and Comparing Structures

A structure is really just a "homemade variable" — so most operations you can do with a regular variable also work with structures. But **two specific operations** need special attention: copying and comparing.

### Copying Structures

Unlike arrays (which can't be assigned with `=`), structures **can** be copied with a simple assignment:

```c
typedef struct {
    int day;
    int month;
    int year;
} date;

int main() {
    date d1 = { .day = 13, .month = 4, .year = 2026 };
    date d2;

    d2 = d1;    // copies ALL members from d1 to d2 — this just works!

    printf("d2 = %d/%d/%d\n", d2.day, d2.month, d2.year);
    return 0;
}
```

**Output:**

```shell
d2 = 13/4/2026
```

The assignment `d2 = d1` copies **every member** from `d1` into `d2` — including any arrays inside the struct. This is one of the few times in C where copying a "collection" with `=` actually works.

### Why Does This Work for Structs But Not Arrays?

Remember from Lab 5 that an array name is really a **pointer to the first element** — it's not a "real" variable you can assign to. A structure, on the other hand, **is** a real variable type. The compiler knows its exact size and layout, so it can copy it byte-for-byte with a single assignment.

### Copying with `memcpy`

For more control (or when copying through pointers), you can use `memcpy` from `<string.h>`:

```c
#include <string.h>

date d1 = { .day = 13, .month = 4, .year = 2026 };
date d2;

memcpy(&d2, &d1, sizeof(date));    // copies sizeof(date) bytes from d1 to d2
```

`memcpy(destination, source, number_of_bytes)` copies raw bytes. For simple structures, `d2 = d1` is cleaner and safer. Use `memcpy` mainly when you're already working with pointers or dealing with dynamically allocated structs.

### Comparing Structures — `==` Does NOT Work

Here's a trap students fall into all the time:

```c
date d1 = { .day = 13, .month = 4, .year = 2026 };
date d2 = { .day = 13, .month = 4, .year = 2026 };

if (d1 == d2) {                    // ⚠️ COMPILE ERROR!
    printf("Equal\n");
}
```

This **won't compile**. C does not support `==` for structures, even though the two structs clearly have identical contents.

### Why Doesn't `==` Work?

Compilers sometimes add **padding bytes** inside structures to align members to memory boundaries for faster access. These padding bytes have **unpredictable** values. Two structures with identical member values might have different padding bytes — so a byte-by-byte comparison would falsely report them as different.

For example:

```c
struct tm1 {
    int x;
    char y;
    // compiler may add 3 bytes of padding here to align the next int
};
```

might have a different memory layout than:

```c
struct tm2 {
    char y;
    int x;
    // padding may be in a different place
};
```

Because of this, C leaves comparison up to **you**.

### Comparing Structures — The Right Way

You have two options:

**Option 1 — Field by field (most common, most reliable):**

```c
int dates_equal(date d1, date d2) {
    return (d1.day == d2.day &&
            d1.month == d2.month &&
            d1.year == d2.year);
}
```

This is the **safest and most explicit** method. You know exactly which fields matter, and you avoid any issues with padding bytes.

**Option 2 — `memcmp` (works only if you're careful):**

`memcmp(ptr1, ptr2, size)` from `<string.h>` compares raw bytes:

```c
#include <string.h>

if (memcmp(&d1, &d2, sizeof(date)) == 0) {
    printf("Equal\n");
}
```

⚠️ **This only works reliably if your struct has no padding** — which you usually can't guarantee. Prefer field-by-field comparison for correctness.

### The `compareDates` Pattern — Like `strcmp`

When comparing two things, a useful pattern is to return `-1`, `0`, or `1` — just like `strcmp` does for strings. This is what C's `qsort` (from Lab 7) expects too.

```c
int compareDates(date d1, date d2) {
    // Compare year first
    if (d1.year < d2.year) return -1;
    if (d1.year > d2.year) return  1;

    // Years equal → compare month
    if (d1.month < d2.month) return -1;
    if (d1.month > d2.month) return  1;

    // Years and months equal → compare day
    if (d1.day < d2.day) return -1;
    if (d1.day > d2.day) return  1;

    return 0;   // fully equal
}
```

**Why this hierarchy?** Dates have a **natural order**: a year difference outweighs a month difference, which outweighs a day difference. The function only falls through to the next field when the current one is equal — this is exactly how you compare multi-part data.

### Usage

```c
int result = compareDates(d1, d2);

if (result < 0) {
    printf("d1 is earlier\n");
} else if (result > 0) {
    printf("d1 is later\n");
} else {
    printf("Same date\n");
}
```

### Summary

| Operation | Works with `=` / `==`? | Correct Approach |
| --------- | ---------------------- | ---------------- |
| Copying a struct | ✓ `d2 = d1` works | `=` or `memcpy` |
| Comparing structs | ✗ `d1 == d2` fails | Field-by-field comparator function |
