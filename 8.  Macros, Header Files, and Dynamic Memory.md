## Warmup ŌĆö Filtering Input with `scanf`

Before we dive into the preprocessor and dynamic memory, let's quickly reconnect to something from last lab. Recall that `scanf` supports **character set format specifiers** for filtering input:

| Specifier | Behavior |
| --------- | -------- |
| `%[abc]` | Accept **only** `a`, `b`, or `c` ŌĆö stop at anything else |
| `%[^abc]` | Accept **everything except** `a`, `b`, or `c` |
| `%[A-Z]` | Accept **only** uppercase letters ŌĆö stop at anything else |
| `%[^\n]` | Accept everything except newline (reads full line) |
You can also specify **ranges** inside the brackets using `-`:
```c
scanf("%[A-Za-z]", str);     // accept only alphabetic characters
scanf("%[0-9]", str);        // accept only digits
scanf("%[A-Za-z0-9]", str);  // accept alphanumeric only
```

**Example ŌĆö reading only digits:**
```c
#include 

int main() {
    char str[100];

    printf("Enter input: ");
    scanf("%[0-9]", str);     // stops at first non-digit

    printf("Digits only: %s\n", str);
    return 0;
}
```

**Output:**
```shell
Enter input: 12345abc
Digits only: 12345
```

The scan stopped at `a` because `a` isn't in the range `0-9`.

## The Preprocessor

The **C Preprocessor** is not part of the compiler ŌĆö it's a separate tool that runs **before** compilation begins. Its job is simple: perform **text substitution** on your source code. Anything starting with `#` is a preprocessor directive, and the preprocessor handles it before the compiler even sees your code.
### What Does It Actually Do?

When you write:
```c
#include <stdio.h>
#define PI 3.14159

int main() {
    printf("Pi = %f\n", PI);
    return 0;
}
```

The preprocessor:
1. **Replaces** `#include <stdio.h>` with the entire contents of `stdio.h`
2. **Substitutes** every occurrence of `PI` with `3.14159`
3. Produces a single expanded file that the compiler then compiles

After preprocessing, your code effectively becomes:
```c
// ... thousands of lines from stdio.h ...

int main() {
    printf("Pi = %f\n", 3.14159);
    return 0;
}
```
### All Preprocessor Directives

| Directive | Purpose |
| --------- | ------- |
| `#define` | Create a macro (value or code substitution) |
| `#include` | Insert contents of another file |
| `#undef` | Remove a previously defined macro |
| `#ifdef` | Compile block **only if** macro is defined |
| `#ifndef` | Compile block **only if** macro is NOT defined |
| `#if` | Compile block if a compile-time condition is true |
| `#else` | Alternative branch for `#if`/`#ifdef` |
| `#elif` | `#else if` ŌĆö another condition to test |
| `#endif` | End a conditional block |
| `#error` | Print error message and stop compilation |
| `#pragma` | Compiler-specific instructions |

**Important:** Preprocessor directives do **not** end with a semicolon ŌĆö they're not C statements.
## Parameterized Macros

We've used `#define` for simple substitutions like `#define PI 3.14159`. But macros can also **take parameters**, acting like tiny inline functions.

### Syntax
```c
#define MACRO_NAME(parameter) expression
```

**Example:**
```c
#define SQUARE(x) ((x) * (x))

int main() {
    int result = SQUARE(5);      // becomes: ((5) * (5)) = 25
    printf("%d\n", result);
    return 0;
}
```

The preprocessor literally replaces `SQUARE(5)` with `((5) * (5))` before the compiler sees it.
### More Examples
```c
#define MAX(a, b) ((a) > (b) ? (a) : (b))
#define MIN(a, b) ((a) < (b) ? (a) : (b))
#define ABS(x)    ((x) < 0 ? -(x) : (x))

int largest = MAX(10, 20);       // becomes: ((10) > (20) ? (10) : (20)) = 20
int smallest = MIN(3, 7);        // becomes: ((3) < (7) ? (3) : (7)) = 3
int positive = ABS(-15);         // becomes: ((-15) < 0 ? -(-15) : (-15)) = 15
```
### ŌÜĀ’ĖÅ Always Wrap Parameters in Parentheses

This is the most common macro bug. Consider this unsafe version:
```c
#define SQUARE(x) x * x          // ŌÜĀ’ĖÅ DANGEROUS ŌĆö no parentheses!

int result = SQUARE(2 + 3);      // becomes: 2 + 3 * 2 + 3 = 11 (not 25!)
```

Because the preprocessor does **pure text substitution**, `2 + 3` gets inserted as-is, and multiplication's higher precedence ruins the math. The fix:
```c
#define SQUARE(x) ((x) * (x))    // safe

int result = SQUARE(2 + 3);      // becomes: ((2 + 3) * (2 + 3)) = 25 Ō£ō
```
### Macro vs Function

| | Macro | Function |
| --- | --- | --- |
| When processed | Before compilation (text substitution) | At runtime (function call) |
| Type checking | **No** ŌĆö any type works | Yes ŌĆö strict types |
| Speed | Faster (no call overhead) | Slower (function call overhead) |
| Debugging | Harder (substituted before compile) | Easier |
| Size impact | Code grows with each use | Code stays small |

Use macros for **simple, fast, type-agnostic** operations. Use functions for **complex logic** where type safety and debugging matter.
## Conditional Compilation

Conditional compilation lets you **include or exclude** blocks of code based on whether a macro is defined or a condition is true. This is hugely useful for debugging, platform-specific code, and configurable builds.
### `#ifdef` and `#ifndef`
```c
#ifdef MACRO_NAME
    // compiled only if MACRO_NAME is defined
#endif

#ifndef MACRO_NAME
    // compiled only if MACRO_NAME is NOT defined
#endif
```

**Classic use case ŌĆö debug prints:**
```c
#include <stdio.h>
#define DEBUG

int main() {
    int x = 42;

#ifdef DEBUG
    printf("DEBUG: x = %d\n", x);
#endif

    printf("Final result: %d\n", x);
    return 0;
}
```

When `DEBUG` is defined, both lines print. When you comment out `#define DEBUG`, the `printf` inside `#ifdef` is **removed by the preprocessor** ŌĆö it's as if that code never existed. No runtime overhead, no need to delete code manually.
### `#if`, `#else`, `#elif`, `#endif`

These work like `if-else-if` chains but at **compile time**:
```c
#define VERSION 2

#if VERSION == 1
    printf("Running version 1\n");
#elif VERSION == 2
    printf("Running version 2\n");
#else
    printf("Unknown version\n");
#endif
```

Only **one** branch is compiled into the final program. The others are completely removed.
### Why Is This Useful?

1. **Debugging toggles** ŌĆö Turn diagnostic output on/off with one line
2. **Platform code** ŌĆö Compile different code for Windows vs Linux
3. **Feature flags** ŌĆö Enable or disable features without deleting code
4. **Device variants** ŌĆö Same source, different builds for different hardware

## Header Files

A **header file** (`.h`) is a file that contains **declarations** ŌĆö function prototypes, macros, type definitions, and global variable declarations ŌĆö that can be shared across multiple source files. Header files are how C programs stay organized as they grow.
### System vs User Header Files

You've been using system headers all along:
```c
#include <stdio.h>      // system header ŌĆö from standard library
#include "myheader.h"   // user header ŌĆö from your own project
```

| Form | Where the Preprocessor Looks |
| ---- | ---------------------------- |
| `#include <...>` | **System directories only** (e.g. `/usr/include`) |
| `#include "..."` | **Current directory first**, then system directories |
Use `< >` for standard library headers and `" "` for your own header files.
### Why Create Your Own Header Files?

As programs grow, putting everything in `main.c` becomes unmanageable. Header files let you:
1. **Share declarations** between multiple `.c` files
2. **Organize related code** into logical units
3. **Hide implementation details** ŌĆö users of a header only need to see the interface
4. **Reuse code** across different projects
### Anatomy of a Header File

A header file typically contains:
- `#define` macros and constants
- `typedef` declarations
- Function **prototypes** (not definitions)
- `struct` and `enum` declarations

A header file **should NOT** contain actual function definitions ŌĆö those go in the corresponding `.c` file.

**Example ŌĆö `mathutils.h`:**
```c
#define PI 3.14159
#define SQUARE(x) ((x) * (x))

int add(int a, int b);
int multiply(int a, int b);
float circleArea(float radius);
```

**Example ŌĆö `mathutils.c`:**
```c
#include "mathutils.h"

int add(int a, int b) {
    return a + b;
}

int multiply(int a, int b) {
    return a * b;
}

float circleArea(float radius) {
    return PI * SQUARE(radius);
}
```

**Example ŌĆö `main.c`:**
```c
#include <stdio.h>
#include "mathutils.h"

int main() {
    printf("3 + 4 = %d\n", add(3, 4));
    printf("5 * 6 = %d\n", multiply(5, 6));
    printf("Area of circle (r=2) = %.2f\n", circleArea(2.0));
    return 0;
}
```
### Compiling Multiple Files

When your project has multiple `.c` files, you must tell `gcc` to compile **all of them** together:
```shell
gcc main.c mathutils.c -o program
./program
```

The compiler compiles each `.c` file separately, then **links** them into a single executable. The `.h` file is **not** passed to `gcc` ŌĆö it gets pulled in automatically by `#include`.
### Include Guards

If a header gets `#include`d multiple times (directly or indirectly), it could cause duplicate declarations ŌĆö which is a compile error. **Include guards** prevent this using conditional compilation:
```c
#ifndef MATHUTILS_H
#define MATHUTILS_H

// ... header contents ...

#endif
```

**How it works:**
1. First time the header is included: `MATHUTILS_H` isn't defined yet, so the block gets processed and `MATHUTILS_H` becomes defined
2. Second (or any later) time: `MATHUTILS_H` is already defined, so the entire block is skipped

This guarantees each header is processed **exactly once** per compilation, no matter how many times it's included.

## Static vs Dynamic Memory

In C, there are **two ways** memory can be allocated for your variables: **static** (decided at compile time) and **dynamic** (decided at runtime). Understanding the difference is crucial for writing flexible programs.
### Static Memory

**Static memory** is reserved **before** the program runs. The compiler looks at your code and decides how much memory each variable needs based on its type and declared size.
```c
int a;                // 4 bytes reserved at compile time
int numbers[100];     // 400 bytes reserved at compile time
char buffer[256];     // 256 bytes reserved at compile time
```

**Characteristics:**
- Size must be **known at compile time**
- Cannot be resized once the program runs
- Automatically managed (no manual cleanup)
- Lives on the **stack** (for locals) or in **data/BSS segments** (for globals)
### The Problem with Static Memory

Say you're writing a program to store student names, but you don't know how many students will enroll:
```c
int students[100];   // reserved 400 bytes
```

But what if:
- Only **12 students** actually enroll? ŌåÆ **352 bytes wasted**
- **150 students** enroll? ŌåÆ **Not enough space, program breaks!**

Static memory forces you to either **over-allocate** (wasting memory) or **under-allocate** (risking crashes). The size is **locked in at compile time**.
### Dynamic Memory

**Dynamic memory** is allocated **at runtime**, meaning your program decides how much memory it needs **while it's running** ŌĆö based on user input, file sizes, or other conditions that aren't known in advance.
```c
int n;
scanf("%d", &n);             // ask user how many
int *arr = malloc(n * sizeof(int));   // allocate exactly n ints
```

**Characteristics:**
- Size determined **at runtime**
- Can be **resized** later (with `realloc`)
- **Must be freed manually** when done (or you get a memory leak)
- Lives on the **heap** (the memory segment for dynamic allocation ŌĆö see Lab 6)
- Accessed **only through pointers**
### Static vs Dynamic ŌĆö Side by Side

|                 | Static Memory            | Dynamic Memory  |
| --------------- | ------------------------ | --------------- |
| When allocated  | Compile time             | Runtime         |
| Size known      | Before running           | During running  |
| Can resize?     | No                       | Yes (`realloc`) |
| Manual cleanup? | No                       | Yes (`free`)    |
| Where it lives  | Stack / Data segment     | Heap            |
| Access          | Variable name or pointer | Pointer only    |
### When to Use Dynamic Memory
- Reading a file whose size you don't know in advance
- Data structures that grow or shrink (linked lists, growing arrays)
- Large objects that won't fit on the stack (> ~2MB)
- Any program where memory needs depend on user input
## `malloc`, `calloc`, and `free`

C provides two main functions for allocating memory at runtime, both from `<stdlib.h>`:
### `malloc` ŌĆö Memory Allocation

**Syntax:**
```c
void *malloc(size_t size);
```

- Takes **one argument**: the number of **bytes** to allocate
- Returns a **pointer to the allocated block**, or `NULL` if allocation fails
- Memory contents are **uninitialized** ŌĆö contain garbage values
- Return type is `void *`, which is typically cast to the desired pointer type

**Example:**
```c
int *arr = (int *)malloc(5 * sizeof(int));    // allocate space for 5 ints
```

The `5 * sizeof(int)` pattern is standard: multiply the number of elements by the size of each element to get total bytes. On most systems this is `5 * 4 = 20` bytes.
### `calloc` ŌĆö Cleared Allocation

**Syntax:**
```c
void *calloc(size_t nitems, size_t size);
```

- Takes **two arguments**: number of items and size of each item
- Returns a pointer to the allocated block, or `NULL` on failure
- **Automatically initializes** all bytes to `0`

**Example:**
```c
int *arr = (int *)calloc(5, sizeof(int));     // allocate 5 ints, all zeroed
```
### `malloc` vs `calloc`

| | `malloc` | `calloc` |
| --- | --- | --- |
| Number of arguments | 1 (total bytes) | 2 (count and size) |
| Initial values | Garbage (uninitialized) | All zeros |
| Speed | Slightly faster | Slightly slower (due to zeroing) |
| Use when | You'll overwrite everything anyway | You need a clean, zeroed block |
### `free` ŌĆö Release the Memory

When you're done using dynamically allocated memory, you **must** give it back to the operating system using `free()`:

**Syntax:**
```c
void free(void *ptr);
```

**Example:**
```c
int *arr = (int *)malloc(10 * sizeof(int));
// ... use arr ...
free(arr);     // release the memory
arr = NULL;    // good practice ŌĆö prevents accidental reuse
```

If you forget to `free`, the memory stays reserved even after you stop using it ŌĆö this is called a **memory leak**. In a long-running program, leaks can eventually consume all available memory and crash the system.
### Complete Example
```c
#include <stdio.h>
#include <stdlib.h>

int main() {
    int n;

    printf("How many numbers? ");
    scanf("%d", &n);

    int *arr = (int *)calloc(n, sizeof(int));    // allocate & zero out

    if (arr == NULL) {                            // always check!
        printf("Memory allocation failed!\n");
        return 1;
    }

    printf("Enter %d numbers:\n", n);
    for (int i = 0; i < n; i++) {
        scanf("%d", &arr[i]);
    }

    printf("You entered: ");
    for (int i = 0; i < n; i++) {
        printf("%d ", arr[i]);
    }
    printf("\n");

    free(arr);        // release memory
    return 0;
}
```
### ŌÜĀ’ĖÅ Common Mistakes

| Mistake | Problem |
| ------- | ------- |
| Forgetting `free()` | **Memory leak** ŌĆö memory stays reserved forever |
| Using pointer after `free()` | **Dangling pointer** ŌĆö undefined behavior, crashes |
| Freeing the same pointer twice | **Double free** ŌĆö crash or corruption |
| Not checking for `NULL` return | Program crashes if allocation fails |
| Mixing up `malloc(n, sizeof(int))` with `calloc` | `malloc` takes **one** argument, not two! |
### Dynamic Memory and 2D Arrays

For a dynamic 2D matrix, you can allocate a **single contiguous block** using the row-major formula from Lab 6:
```c
int rows = 3, cols = 3;
int *matrix = (int *)calloc(rows * cols, sizeof(int));

// Access element at row i, column j using flattened indexing:
// matrix[i * cols + j]   ŌåÉ   same as matrix[i][j] for a static 2D array

matrix[0 * cols + 0] = 1;    // top-left
matrix[2 * cols + 2] = 9;    // bottom-right
```

This is the same formula you saw for passing 2D arrays as flattened pointers: `BaseAddress + (i ├Ś NumCols + j) ├Ś SizeOfType`. The beauty of dynamic allocation is that `rows` and `cols` can come from user input at runtime.

## `realloc` ŌĆö Resizing Allocated Memory

What if you allocate 10 slots with `malloc`, fill them up, and then need more? You can't just extend a `malloc`'d block in place. That's where `realloc` comes in ŌĆö it **resizes** an existing memory block.
### Syntax
```c
void *realloc(void *ptr, size_t new_size);
```

| Parameter | Meaning |
| --------- | ------- |
| `ptr` | Pointer to a previously allocated block (from `malloc`, `calloc`, or `realloc`) |
| `new_size` | New size in **bytes** for the block |
### How It Works

`realloc` attempts to resize the block pointed to by `ptr`:
1. If there's **space available** right after the current block, it simply extends it in place
2. If not, it **allocates a new block** of the requested size, **copies** the existing data over, and **frees** the old block
3. Returns a pointer to the (possibly new) block, or `NULL` if allocation fails

**The returned pointer may be different from the original**, so always assign the result back:
```c
int *arr = (int *)malloc(5 * sizeof(int));
// ... fill arr with 5 values ...

arr = (int *)realloc(arr, 10 * sizeof(int));   // grow to 10 ints
```
### Special Cases

| Call | Behavior |
| ---- | -------- |
| `realloc(NULL, size)` | Equivalent to `malloc(size)` ŌĆö allocates a new block |
| `realloc(ptr, 0)` | Frees `ptr` and returns `NULL` |
| `realloc(ptr, smaller_size)` | Shrinks the block, data beyond the new size is lost |
| `realloc(ptr, larger_size)` | Grows the block, new bytes are **uninitialized** |
### Example
```c
#include <stdio.h>
#include <stdlib.h>
#include <string.h>

int main() {
    char *str = (char *)malloc(15);
    strcpy(str, "Hello");
    printf("Before: %s  (Address: %p)\n", str, (void *)str);

    // Need more space ŌĆö grow the block
    str = (char *)realloc(str, 30);
    strcat(str, ", World!");
    printf("After:  %s  (Address: %p)\n", str, (void *)str);

    free(str);
    return 0;
}
```

**Output:**
```shell
Before: Hello  (Address: 0x55a7c8e2e2a0)
After:  Hello, World!  (Address: 0x55a7c8e2e2a0)
```

The address may or may not change depending on whether `realloc` could extend in place.
### ŌÜĀ’ĖÅ Safe `realloc` Pattern

If `realloc` fails, it returns `NULL` ŌĆö but the **original block is still valid**. If you overwrite your only pointer before checking, you've just lost access to your data (memory leak):
```c
// ŌÜĀ’ĖÅ Unsafe:
arr = realloc(arr, new_size);    // if realloc fails, arr becomes NULL and old data is lost forever

// Ō£ō Safer:
int *temp = realloc(arr, new_size);
if (temp == NULL) {
    printf("Reallocation failed\n");
    // arr is still valid here ŌĆö handle the error
} else {
    arr = temp;    // only overwrite if realloc succeeded
}
```

### Summary ŌĆö Dynamic Memory Functions

| Function | Purpose | Initializes Memory? |
| -------- | ------- | ------------------- |
| `malloc(size)` | Allocate `size` bytes | No (garbage values) |
| `calloc(n, size)` | Allocate `n ├Ś size` bytes | Yes (all zeros) |
| `realloc(ptr, size)` | Resize existing block | No (new bytes uninitialized) |
| `free(ptr)` | Release memory | N/A |
